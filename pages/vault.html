Vault

A self-contained, offline-first password manager — AES-256-GCM encrypted in the browser, zero backend, zero dependencies, no data ever leaves the page.

GLJ-Tracker All Projects
🔒

The Problem

Cloud password managers require trust in a third-party server. Browser-built-ins leak metadata. A local solution that encrypts data before it ever touches storage — and that never phones home — is the only truly private option.

🗄️

What It Is

A single vault.html file that is both the app and the database. All account data lives inside the file itself as an AES-256-GCM encrypted blob. Open it in any browser, enter your master password, done.

🎯

The Goal

No installs, no server, no account. One file you back up yourself. Full CRUD for credentials, TOTP-style password-age timers, CSV import from Firefox, and encrypted export — all in <1 400 lines of vanilla HTML.

Architecture

One file, self-contained

There is no backend. The encrypted vault blob is embedded directly in the HTML file. Every operation — unlock, render, edit, export — happens entirely in the browser via the WebCrypto API.

Three views

  • 1 Lock screen — master password prompt. First use initialises an empty vault; any subsequent use must decrypt the existing blob.
  • 2 Dashboard — card grid of all entries. Search with live highlight, copy username/password, reveal/mask toggle, password-age timer.
  • 3 Editor — full CRUD for entries plus CSV import and encrypted export back to the HTML file.

Cryptography

AES-256-GCM via WebCrypto

Everything runs through the browser's native crypto.subtle API — no third-party crypto library, no JS reimplementation.

key derivation 
// PBKDF2: password + random salt → AES key
async function deriveKey(password, salt) {
  const enc = new TextEncoder();
  const km = await crypto.subtle.importKey(
    'raw', enc.encode(password),
    'PBKDF2', false, ['deriveKey']
  );
  return crypto.subtle.deriveKey({
    name: 'PBKDF2',
    salt,
    iterations: 310000,
    hash: 'SHA-256'
  }, km, { name: 'AES-GCM', length: 256 },
  false, ['encrypt', 'decrypt']);
}

Crypto spec

  • KDF PBKDF2-SHA-256 with 310 000 iterations and a 16-byte random salt generated at first use. The salt is stored alongside the ciphertext.
  • ENC AES-256-GCM with a fresh 12-byte IV per save. GCM provides both confidentiality and authenticated integrity — a tampered blob will fail to decrypt.
  • OUT Salt + IV + ciphertext are base64-encoded into a single VAULT_DATA constant inside the HTML file — the file is its own encrypted database.
  • KEY The derived key is held in memory only for the session. It is never serialised, never written to localStorage, never sent anywhere.

Features

What's inside the vault

Each entry stores name, icon, tag, username, password, URL, free-text notes, and a password-change period for the age timer.

Feature Detail
Search & highlight Live filter across name, username, URL, and notes. Matching substrings are highlighted with <mark>.
Copy to clipboard One-click copy for username and password. Passwords shown as dots by default; a reveal toggle unmasks them.
Password-age timer Each entry tracks when the password was last changed. A coloured progress bar (green → yellow → red) shows how stale it is relative to a configurable period.
CSV import Drag-and-drop or browse for a Firefox Lockwise CSV export. Duplicate detection by URL + login. Choose append or replace mode.
Encrypted export Re-encrypts the current vault and outputs the blob — paste it back into the HTML as the new VAULT_DATA to persist your changes.
Offline / no CDN The only external resource is the Tabler icon font. The crypto and app logic are fully self-contained — works on a USB stick with no internet.

Security model

What it protects — and what it doesn't

Vault is designed for a specific threat model: keeping credentials safe at rest in a portable file. It is not a hardened secret store — understand the limits.

✅ At-rest encryption
All data is encrypted before it is written. A stolen copy of the file is useless without the master password.
✅ No network exposure
Nothing leaves the browser. No analytics, no telemetry, no sync endpoint, no update check.
✅ Authenticated encryption
GCM's built-in authentication tag means a corrupted or tampered blob will fail to decrypt rather than silently returning garbage.
⚠️ Master password strength
Security is only as good as the master password. A weak password with 310k PBKDF2 rounds is still a weak password.
⚠️ Browser memory
Decrypted data lives in JS heap while the vault is unlocked. A malicious browser extension with script access could read it.
⚠️ No 2FA / biometrics
The only gate is the master password. There is no second factor, no hardware key, no session timeout — locking is manual.

Usage

Getting started

No install, no build step. Open the file, set a master password, add entries, export to save.

First time

  • 1 Open vault.html in any modern browser.
  • 2 Enter any password — this becomes your master password and initialises an empty vault.
  • 3 Click Edit vault, then + Add to create your first entry.
  • 4 Go to Export, click Generate, copy the blob, and paste it into vault.html as the value of const VAULT_DATA. Save the file.
  • 5 Back up the file wherever you like — USB stick, Dropbox, NAS. The encryption travels with it.

Importing from Firefox

  • 1 In Firefox, go to about:logins → ⋯ menu → Export logins. Save the CSV.
  • 2 Unlock Vault, click Edit vault, scroll to Import CSV.
  • 3 Drop the file or click to browse. The preview shows how many entries were found, how many are duplicates (matched by URL + login), and how many are new.
  • 4 Choose Append (merge) or Replace (overwrite entire vault). Then export and save as above.

Note  The exported Firefox CSV is plaintext — delete it from your Downloads folder after importing.
