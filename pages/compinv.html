Component Inventory

A desktop app for tracking a physical electronics parts collection — searchable, sortable, CSV-importable, and packaged as a portable Windows executable.

GitHub ↗ Arachnid Cabinet All Projects
📦

The Problem

Six physical boxes and a bag of electronics parts, each with dozens of components across different categories and sub-locations. A markdown file worked briefly — until finding a specific transistor or checking stock of a capacitor meant reading through hundreds of lines.

⚙️

What It Is

A Python/tkinter desktop application backed by SQLite. It presents a sortable, searchable table of every component with fields for name, value, quantity, physical container, category, sublocation, datasheet, notes, and a configurable low-stock threshold. Packaged as a portable .exe via PyInstaller — no install needed.

🎯

The Goal

Answer "do I have one and where is it?" in under five seconds. Full-text search across all fields, per-field filtering, clickable column headers for sorting, and a one-click low-stock view that surfaces anything running low before starting a project.

Stack

Technology choices

Deliberately minimal — no web server, no framework, no dependency on internet access. The entire application is one Python file and a SQLite database that lives next to the executable.

Portability

The get_db_path() function detects whether the process is a frozen PyInstaller bundle (sys.frozen) or a plain Python script and places inventory.db next to the executable in either case. Move the dist/ folder anywhere — the database travels with it.

Database

Schema

Two tables. components holds every part; settings is a simple key/value store for future app preferences. Both are created on first launch via CREATE TABLE IF NOT EXISTS.

components table

ColumnTypePurpose
id INTEGER PKAuto-incrementing row identifier
name TEXT NOT NULLComponent name — the only required field
value TEXTElectrical value, part number, or spec
quantity INTEGERCurrent stock count
container TEXTPhysical box or bag label
category TEXTType grouping within the container
sublocation TEXTNested bag or compartment within the category
datasheet TEXTLocal file path or URL to datasheet
notes TEXTFree-text notes and observations
low_stock INTEGERAlert threshold; 0 = no alert
fetch_all() — search query 
SELECT * FROM components
WHERE
  name      LIKE ?
  OR value    LIKE ?
  OR container LIKE ?
  OR category  LIKE ?
  OR sublocation LIKE ?
  OR notes    LIKE ?
-- AND low_stock > 0 AND quantity <= low_stock
-- (appended in low-stock mode)
ORDER BY container, category, name

settings table

A key/value store using INSERT OR REPLACE — upsert semantics without requiring a unique constraint migration. Reserved for future persistent preferences such as last-used filter or window geometry.

User Interface

Features

Three horizontal zones — a dark toolbar, a search bar, and the component table — plus a slim status bar at the bottom showing the active database path.

CRUD toolbar

Add, Edit, Delete, and Clear All buttons in a navy toolbar. Edit also opens on double-click; Delete fires on the Delete key. Clear All requires a confirmation dialog.

Add / Edit dialog

A modal window with one field per column — entries, spinboxes for numeric values, a multi-line text area for notes, and Browse / Open buttons for the datasheet path. The dialog blocks the main window until dismissed.

Live search

The table refilters on every keystroke via a StringVar trace. A field dropdown narrows the search to a single column (Name, Value, Container, Category, Sublocation, or Notes). A Clear button resets both.

Column sort

Clicking any column header sorts ascending; clicking again reverses. Quantity is sorted numerically; all other columns alphabetically (case-insensitive). Sort state persists across search filter changes.

Low-stock mode

A purple toolbar button toggles a filter that shows only rows where quantity ≤ low_stock and low_stock > 0. The button visually depresses (sunken relief) while active.

Row highlighting

Low-stock rows are tinted rose-red with crimson text. Even-numbered non-low rows get a faint blue-white alternating stripe. Selected rows turn indigo with white text.

CSV import

Opens a file picker, then asks Replace or Append. Reads utf-8-sig (BOM-stripped) — handles files saved from Excel without garbled characters. Skips rows with no name.

CSV export

Exports the current filtered view — not always the full database. If a search or low-stock filter is active, only those rows are written. File is utf-8-sig for Excel compatibility.

Data Pipeline

Markdown → CSV → Database

The physical inventory was originally maintained as a structured markdown file. A two-stage Python pipeline converted it into the app's SQLite database without manual re-entry.

Stage 1 — parse_inventory_md.py

  • 1 Reads inventory.md line by line, tracking the current container (Box heading), category (## Category:), and sublocation (nested bag)
  • 2 Each list item is split on - or --- separators; trailing parentheticals become the value field
  • 3 Quantities embedded in names (e.g. - AHT10 - 2 - notes) are parsed into the quantity column
  • 4 Outputs inventory_import.csv with all nine importable fields, ready for the app

Stage 2 — fix_qty.py

  • 1 Reads inventory_import.csv and detects three quantity patterns still embedded in name strings: trailing ×N, Bag: Nx …, and leading Nx …
  • 2 Smart disambiguation — skips 8×8 and 172×320 (no space before ×) and flags compound bags (Bag: 3x A + 10x B) for manual review
  • 3 Only applies changes when the existing quantity is 0 — never overwrites a value already set by Stage 1
  • 4 Writes inventory_import_fixed.csv and a full inventory_fix_report.txt change log for review before importing

Physical Layout

Box structure

The inventory spans six physical boxes and a tool bag, each containing parts grouped by category and further split into nested bags or compartments stored in the sublocation field.

Box 1 — Modules

Power modules, step-up/buck converters, charging boards, timer relays, remote RF modules, ESP32/Arduino development boards, and sound modules.

Box 2 — Sensors & Kits

Breadboards, protoboards, Arduino Uno, sensor bags (DHT22, MPU-6050, magnetometer, microphones), robot kits, and a temperature prototype.

Box 3 — Components

Resistors, capacitors, diodes, transistors, MOSFETs, NE555 timers, displays (OLED, TFT, LCD), relays, potentiometers, switches, and connectors.

Box 4 — Cables & Mechanical

USB cables, battery holders, PC fans, pin headers, jumper wires, hook-up wire, test leads, barrel jack splitters, servo mounts, and an aluminium chassis.

Box 5 — Light, Water & Motors

WS2812B LED strips, laser modules, UV LEDs, ultrasonic foggers, water pumps, servo motors, ESCs, stepper drivers, H-bridge modules, and PWM speed controllers.

Tool Bag — PSUs

Nine AC-DC power supplies ranging from 5 V / 0.5 A to 12 V / 6 A in mini, medium, and large form factors, plus socket adapters.

Box 6 — Homebox

House repair items, assorted brass wire (0.7 / 1 / 2 mm straight and wound), and brass sheet stock — kept separate from electronics.
